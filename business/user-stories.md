# user-stories.md  

## Epic 1 – Structured Learning Path  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|---------------------|------------|
| 1 | **As a beginner system‑admin student, I want a curated curriculum that starts with fundamentals and builds to advanced topics, so that I can follow a logical progression without feeling lost.** | - Curriculum is displayed as a linear roadmap with clearly labeled modules (e.g., “Linux Basics → Shell Scripting → Networking → Security”). <br>- Each module shows prerequisite modules and estimated completion time. <br>- Users can’t start a module until all prerequisites are marked complete. <br>- A “Jump to next module” button appears only after the current module is finished. <br>- The roadmap is responsive on desktop and mobile. | M |
| 2 | **As a returning learner, I want to resume where I left off, so that I don’t waste time re‑doing content I’ve already completed.** | - System automatically records the last completed lesson per user. <br>- On login, a “Continue Learning” banner takes the user to the next unfinished lesson. <br>- Users can manually set a “Start From” point within the roadmap. <br>- Progress is persisted across devices and browsers. | S |
| 3 | **As an instructor, I want to add, reorder, or retire modules without breaking existing learner progress, so that the curriculum stays up‑to‑date.** | - Admin UI allows drag‑and‑drop reordering of modules. <br>- Adding a new module inserts it at a chosen position and automatically updates prerequisite chains. <br>- Retiring a module flags it as “Deprecated” and hides it from new learners while preserving it for existing learners’ history. <br>- All changes trigger a version bump and are logged for audit. | L |

## Epic 2 – Interactive Labs & Hands‑On Exercises  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|---------------------|------------|
| 4 | **As a learner, I want browser‑based, sandboxed Linux terminals for each lab, so that I can practice commands without installing anything locally.** | - Each lab launches an isolated Docker container accessible via a web terminal (e.g., xterm.js). <br>- Container resets to a clean state on lab start and on “Reset Lab” action. <br>- Terminal supports copy‑paste, autocomplete, and basic file upload/download. <br>- Lab environment matches the OS version specified in the curriculum (e.g., Ubuntu 22.04). <br>- Session expires after 60 min of inactivity. | L |
| 5 | **As a learner, I want automatic validation of lab tasks, so that I receive immediate feedback on whether I completed the steps correctly.** | - Each lab includes a set of hidden test scripts that run on the container after the learner submits. <br>- Test results are displayed as pass/fail with concise hints for failures. <br>- Learners can view a “Solution” after three failed attempts. <br>- Validation runs within 5 seconds of submission. <br>- All test scripts are sandboxed and cannot affect host resources. | M |
| 6 | **As a learner, I want to download a PDF cheat‑sheet for each lab, so that I can reference commands offline.** | - A “Download Cheat‑Sheet” button generates a PDF containing all required commands, explanations, and example outputs. <br>- PDF is dynamically populated based on the current lab version. <br>- File size ≤ 200 KB and is accessible on all major browsers. <br>- Download link expires after 24 hours for security. | S |

## Epic 3 – Progress Tracking, Badges & Certification  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|---------------------|------------|
| 7 | **As a learner, I want a personal dashboard that shows my overall progress, completed labs, and upcoming milestones, so that I can gauge my readiness.** | - Dashboard displays % of curriculum completed, number of labs passed, and next milestone date. <br>- Visual progress bar for each module. <br>- List of earned badges with hover‑over descriptions. <br>- “Export Report” button creates a CSV/JSON summary. <br>- Data updates in real‑time after each lesson completion. | M |
| 8 | **As a learner, I want to earn digital badges for key achievements (e.g., “Linux Fundamentals”, “Network Security”), so that I can showcase my skills on LinkedIn or a résumé.** | - Badges are awarded automatically when predefined criteria are met (e.g., 100 % lab pass rate for a module). <br>- Badge metadata includes title, description, issue date, and verification URL. <br>- Users can download an SVG/PNG badge and a signed JSON credential. <br>- Badges are visible on the dashboard and can be shared via a “Share on LinkedIn” button. | M |
| 9 | **As a graduate, I want to receive a verifiable certificate after completing the full track, so that employers can trust my competency.** | - Certificate is generated as a PDF with the learner’s name, completion date, and a unique verification code. <br>- Verification code links to a public endpoint that displays the learner’s achievement status. <br>- Certificate template is brandable by the admin. <br>- Learner can request re‑issue of the certificate at any time. <br>- Certificate issuance is logged for compliance. | L |

## Epic 4 – Community, Support & Personalization  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|---------------------|------------|
| 10 | **As a learner, I want an integrated discussion thread for each lesson, so that I can ask questions and see peer answers without leaving the platform.** | - Each lesson page has a collapsible comment section powered by a moderated forum. <br>- Users can up‑vote/down‑vote answers; top‑voted answer is highlighted. <br>- Notifications appear when someone replies to a user’s comment. <br>- Moderation tools allow admins to flag/remove inappropriate content. <br>- Thread persists even after the lesson is archived. | M |
| 11 | **As a learner, I want AI‑driven hints that adapt to my mistakes, so that I receive personalized guidance without exposing my code publicly.** | - When a lab validation fails, the system analyses the error and suggests a context‑aware hint (e.g., “Check the permissions of /etc/hosts”). <br>- Hints are limited to three per lab attempt to encourage problem‑solving. <br>- Hints are generated using the internal LLM trained on the lab dataset. <br>- Users can rate the usefulness of each hint (thumbs up/down). <br>- Hint logs are stored for future model improvement. | L |
| 12 | **As a support admin, I want to view a real‑time heatmap of where learners drop off, so that I can prioritize content improvements.** | - Heatmap visualizes drop‑off percentages per lesson/module on the admin dashboard. <br>- Data refreshes every 15 minutes. <br>- Clicking a segment drills down to individual learner sessions (anonymized). <br>- Export button provides CSV for further analysis. <br>- Access restricted to users with “Support” role. | S |

---  

*All stories are written in Connextra format, include 3‑5 acceptance criteria, and are sized for sprint planning (S = small, M = medium, L = large).*
